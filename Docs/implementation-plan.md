## 10. 次フェーズ計画: LINE候補UIの運用化

E2E は完走できたけん、次フェーズは「人が LINE で見て迷わず選べること」と「選ばれた候補がそのまま正しく投稿されること」を優先して仕上げる。

### 目的

- LINE 上で候補をカード形式で見比べられるようにする
- 左右スクロールで複数候補を比較できるようにする
- `post` で選ばれた候補本文が、その候補IDに紐づく内容のまま X に投稿されることを保証する
- 投稿前後の確認メッセージを増やし、運用者が状態を見失わないようにする
- 候補は都度使い捨てにせず保存し、再利用できる運用に寄せる

### LINE UI仕様

- 候補表示は、1候補を1カードとする横スクロール形式のカルーセルUIを採用する
- 運用者は左にスワイプして次の候補を確認できるようにする
- 各候補カードには `型`、`フック`、`本文要約`、選択用ボタン `これにする！` を配置する
- 候補カードには `保留` や `再生成` を持たせず、「この案を選ぶ」ことに役割を絞る
- 候補カルーセルの最後には全体操作用カードを1枚置き、`再生成する` と `今回は投稿しない` をまとめる

### LINE起点トリガー

- 運用者が LINE 上で `種まきだ！` または `収穫だ！` と送信した場合、それを候補生成開始のトリガーとして扱う
- `POST /webhooks/line` では postback だけでなく message event も受け取り、本文に応じて候補生成フローを起動する
- `種まきだ！` は営業系候補生成モードとし、以下の曜日ローテーションで投稿型 ①〜⑦ を切り替える
- 月曜: `気づき喚起型`
- 火曜: `残業訴求型`
- 水曜: `Before/After型`
- 木曜: `問いかけ型`
- 金曜: `共感型`
- 土曜: `CTA型`
- 日曜: `制約型`
- `収穫だ！` は拡散系候補生成モードとし、投稿型 ⑧ 拡散特化型の candidate-delivery を実行する
- 候補生成後はそのまま LINE カルーセルを返し、運用者は LINE 内で比較・選択・投稿確認まで完結できるようにする
- API 直叩きがなくても、日々の運用は `種まきだ！` と `収穫だ！` の2つの文言から開始できる状態を目指す

### 選択後の確認フロー

- 運用者が候補カード内の `これにする！` を押した場合は、即時投稿せず確認ステップへ進む
- 確認メッセージでは、選択された候補の `フック` と `本文` を明示した上で `投稿しますか？` と表示する
- 確認メッセージの操作は `はい` と `やめる` の2択とする
- `はい` が押された場合のみ投稿処理へ進む
- `やめる` が押された場合は投稿を中止し、候補選択に戻れるようにする

### 正確投稿の原則

- 投稿処理は必ず `candidateId` を正本として実行する
- LINE 上に表示された見た目文面ではなく、選択時に保持した `candidateId` に紐づく候補データを DynamoDB から取得して X 投稿に使用する
- これにより、LINE 上で選ばれた候補と実際に投稿される本文の不一致を防ぐ
- あわせて `candidateId -> postId -> externalPostId` の追跡可能性を維持する

### 候補の保存と再利用方針

- 候補生成で作られた案は毎回 DynamoDB の `candidates` に保存する
- 生成済み候補は `generated / sent_to_line / held / discarded / posted` などの状態で管理する
- 候補生成で作られた案は毎回 DynamoDB の `candidates` に保存するが、LINE 上では複雑な整理操作を要求しない
- 1回 `これにする！` が押された時点で、その候補束は締める
- 同じ候補束の他候補は、あとから古いカードを遡っても選べないようにする
- 古いカードから `これにする！` が押された場合は、対象候補がすでに無効化されとることを返し、再生成を促す
- これにより、LINE 上での主操作は `これにする！`、`再生成する`、`今回は投稿しない` の3つに絞る
- 候補の内部保存は続けるが、再利用は運用者が古いカードを遡って行うのではなく、必要時に再生成または再送で扱う

### OpenAI生成仕様

- `種まきだ！` と `収穫だ！` の候補生成数はどちらも `5本` に固定する
- OpenAI からの出力は自由文ではなく、`hook` と `body` を持つ JSON 配列で構造化して受け取る
- `body` は X 向けに短く、改行を多めに入れて読みやすくする
- すべての候補本文の末尾には必ず `https://ph4k.aokigk.com/landing` を付与する
- `種まきだ！` 用の prompt では、BtoB の現場担当者に刺さる営業系投稿として以下を守る
- 投稿先は X
- 読み手は現場で監査・点検・報告をしている担当者
- 売り込み感は弱くする
- フック最重要
- 5本とも切り口を変える
- 型ごとの役割は以下を明示する
- 気づき喚起型: 「あ、それ自分や」と思わせる
- 残業訴求型: しんどさ・負担を強めに出す
- Before/After型: シンプルに変化を見せる
- 問いかけ型: 質問中心で構成する
- 共感型: 「自分だけじゃない」と思わせる
- CTA型: 少し話を聞きたい、で終える
- 制約型: 高い・難しい・使われない等を入れる
- `収穫だ！` 用の prompt では、拡散特化型として以下を守る
- 第三者も「それおかしくない？」と反応できる内容にする
- 違和感・あるある・ツッコミを重視する
- 外部の人でも理解できる表現にする
- 専門用語だらけや内輪ネタを避ける
- 5本とも切り口を変える

### 課題起点での8型の位置づけ

- 8つの型は別々の課題を扱うためのものではなく、同じユーザー課題を異なる角度から伝えるための表現パターンとして扱う
- 基本となるユーザー課題は、`現場で行った監査の記録・共有・報告が分断されており、帰社後の再整理と報告書作成が大きな負担になっていること` と定義する
- 投稿生成では、まずこの課題を固定し、その上でどの角度から刺すかを `型` で切り替える

### 8型ごとの課題の見せ方

- `気づき喚起型`: 「それ、当たり前と思っとったけど実は変やない？」と気づかせる
- `残業訴求型`: 帰社後作業、写真整理、報告書作成のしんどさを強く出す
- `Before/After型`: 導入前は分断・手戻り、導入後は現場完結という変化を見せる
- `問いかけ型`: 「まだ帰社後に写真整理しよる？」「報告書作成まで現場で終わらん？」のように質問で自分ごと化させる
- `共感型`: 「それ自分だけやない」「多くの現場が同じことで困っとる」と感じさせる
- `CTA型`: 課題を示したうえで「少し話を聞いてみたい」で終える
- `制約型`: 「高いツールは入れにくい」「複雑な初期設定は回らん」など導入障壁を先に出したうえで課題に繋げる
- `拡散特化型`: 第三者から見ても「なんでまだそんな運用なん？」と思う違和感として再構成する

### 8型ごとの prompt 固定ルール

- 各型ごとに `hookで何を起こすか`、`bodyを何段で組むか`、`避ける言い回し` を明示し、OpenAI 出力のブレを減らす

#### 気づき喚起型

- hookで何を起こすか: 「それ普通と思っとったけど、実はムダかもしれん」と気づかせる
- bodyを何段で組むか:
- 1段目: 現場で普通にやっている行動を出す
- 2段目: その裏で発生しているムダを指摘する
- 3段目: 残業や手戻りにつながると示す
- 4段目: 現場完結の必要性をにおわせる
- 避ける言い回し:
- 「革命的」
- 「劇的改善」
- 「誰でも簡単」

#### 残業訴求型

- hookで何を起こすか: 帰社後作業のしんどさを一瞬で思い出させる
- bodyを何段で組むか:
- 1段目: 帰社後にやる作業を具体的に列挙する
- 2段目: そこに時間が吸われることを示す
- 3段目: 本来の仕事ではない感覚を出す
- 4段目: 減らせる余地があると締める
- 避ける言い回し:
- 「ブラック」
- 「地獄」
- 大げさな被害表現

#### Before/After型

- hookで何を起こすか: 分断運用と現場完結の差をひと目で理解させる
- bodyを何段で組むか:
- 1段目: Before を短く示す
- 2段目: After を短く示す
- 3段目: 何が減ったかを明示する
- 4段目: 現場完結の価値を一言で締める
- 避ける言い回し:
- 比較が曖昧な表現
- 数字のない誇張
- 説明しすぎる前置き

#### 問いかけ型

- hookで何を起こすか: 自分ごととして答えたくなる状態を作る
- bodyを何段で組むか:
- 1段目: 質問
- 2段目: もう1つ質問
- 3段目: その問いの背景にある課題を明示する
- 4段目: 改善余地を短く示す
- 避ける言い回し:
- 弱い質問
- 抽象的な問い
- 問いだけで終わる構成

#### 共感型

- hookで何を起こすか: 「それ自分だけやない」と安心させる
- bodyを何段で組むか:
- 1段目: よくある現場の状況を描く
- 2段目: 同じ悩みが多いと伝える
- 3段目: 個人の問題ではなく構造の問題だと示す
- 4段目: 変えられる可能性を出す
- 避ける言い回し:
- 上から目線
- 説教っぽい言い回し
- 根拠のない断定

#### CTA型

- hookで何を起こすか: 読み手に「少し見てみようかな」と思わせる
- bodyを何段で組むか:
- 1段目: 課題を短く再提示する
- 2段目: 改善後のイメージを見せる
- 3段目: 強すぎない CTA を置く
- 4段目: LP リンクへ自然につなぐ
- 避ける言い回し:
- 「今すぐ」
- 「絶対」
- 強い押し売り表現

#### 制約型

- hookで何を起こすか: 現実的な導入障壁を認めて警戒を下げる
- bodyを何段で組むか:
- 1段目: 制約を先に言う
- 2段目: そのせいで分断運用が続くと示す
- 3段目: でも現場完結したいニーズはあると返す
- 4段目: 軽さや使いやすさをにおわせる
- 避ける言い回し:
- 他社批判
- 高機能すぎる印象
- 解決を断言しすぎる表現

#### 拡散特化型

- hookで何を起こすか: 第三者も「それ変やね」と反応したくなる違和感を作る
- bodyを何段で組むか:
- 1段目: 一見普通の運用を書く
- 2段目: でも冷静に見ると変だと示す
- 3段目: 非効率や時代遅れ感を短く出す
- 4段目: 余韻を残して終える
- 避ける言い回し:
- 専門用語だらけ
- 内輪ネタ
- 弱いツッコミ

### 日替わり型運用の方針

- 営業系の投稿型 ①〜⑦ は、日ごとに1種類へ固定して運用する案を優先検討する
- 例として、ある日は `気づき喚起型` のみ、別日は `残業訴求型` のみ、という形で日替わりローテーションする
- 拡散特化型 ⑧ は別枠として毎日1本出す運用も選べるようにする
- 1日の投稿構成は `営業2本 + 拡散1本` を基本とし、営業2本はその日のテーマ型から選ぶ前提で設計する
- この運用により、型ごとの比較・分析・改善がしやすくなる

### 期待する体験

- LINE 上の運用体験は `候補を見る`、`1つ選ぶ`、`確認する`、`投稿する` の4段階に整理する
- UI の見やすさと運用の安全性を両立し、誤操作による意図しない投稿を防ぎやすい構成とする

### 実装方針

- LINE 送信は単純な text + quick reply から、カルーセル形式の Flex Message または template message へ切り替える
- 1候補を1カードとして扱い、`型 / フック / 本文冒頭 / candidateId` をカード内に表示する
- カード下部に `投稿 / 保留 / 破棄 / 再生成` の action を置く
- action の postback data には必ず `candidateId` を含め、Webhook 側ではその candidateId を正本として処理する
- `post` 選択時は「選択した candidateId」「投稿対象本文」「X投稿結果」をログと DynamoDB 上で追跡できるようにする

### 検討事項

1. LINE カードUI
- 1画面に複数候補を横並び表示できる message type を採用する
- 長文は全文表示ではなく要約または先頭数行表示にして、詳細は必要なら別 action で見せる
- 候補の順序は生成順または優先順で固定する

2. 選択後の確認導線
- `post` 選択後は LINE に確認メッセージを返す
- 返却内容は `candidateId / hook / 投稿開始 / 投稿完了 or 失敗` を最低限含める
- `hold / discard / regenerate` でも結果を LINE に返し、操作成否をその場で見せる

3. 正確投稿の保証
- `candidateId` をキーに候補本文を取得し、Webhook payload 内の見た目文面は信用しない
- 投稿前に `candidate.status` と `selected` を更新し、投稿後の `post.candidateId` と対応付ける
- CloudWatch Logs と DynamoDB 上で `candidateId -> postId -> externalPostId` を追跡できる状態を維持する

4. 運用テスト
- 候補3件以上を LINE で横スクロールできるか確認する
- 2件以上の候補から任意の1件を選んだとき、意図した candidateId が投稿されるか確認する
- 保留 / 破棄 / 再生成の各 action が誤投稿なく処理されるか確認する

### 推奨の進め順

1. LINE メッセージをカルーセルカードへ変更
2. Webhook 選択後の確認メッセージ追加
3. `candidateId` ベースの正確投稿確認ログを強化
4. 実運用想定で LINE 上から候補選択テストを実施

## Click Tracking 方針

### 目的

- X 側の `non_public_metrics.url_link_clicks` に依存せず、投稿ごとのリンククリック数を自前で正確に計測する
- `postId / candidateId / type / mode` と LP 流入をひも付け、スプレッドシート分析へ活用できるようにする
- URL を長いクエリ付き LP リンクにせず、短い見た目のまま計測可能にする

### 最小構成

1. 投稿本文には LP 直リンクではなく、短い計測用 URL を入れる
- 形式は `https://ph4k.aokigk.com/r/{shortId}` とする
- X 上で見えるリンクは短く保つ

2. 計測用 URL へアクセスが来たら click log を保存する
- 保存対象は最低限以下とする
- `shortId`
- `clickedAt`
- `userAgent`
- `referer`
- 必要なら `ip`

3. 保存後、LP へ `302 redirect` する
- 遷移先は `https://ph4k.aokigk.com/landing`
- 将来的に LP URL が変わっても mapping 側だけ差し替えられる構成にする

4. `shortId` と投稿情報を事前にひも付けておく
- `shortId -> postId / candidateId / type / mode / landingUrl`
- これにより、どの投稿経由の click かを確実に分析できる

### 推奨データ

- click mapping
- `shortId`
- `postId`
- `candidateId`
- `type`
- `mode`
- `landingUrl`
- `createdAt`

- click events
- `shortId`
- `clickedAt`
- `userAgent`
- `referer`
- `ip` または匿名化済みIP

### 運用上の考え方

- X の `URL clicks` は補助指標とし、正本は自前 click tracking に置く
- スプレッドシート分析では `X metrics` と `自前 click 数` を並べて見る
- 将来的に Instagram や LINE など他流入へも横展開しやすいよう、`source` を持てる設計にしてよい

### 実装順

1. redirect 用 endpoint を追加する
2. `shortId` と投稿情報の mapping 保存処理を追加する
3. 投稿本文の LP リンクを `https://ph4k.aokigk.com/r/{shortId}` へ差し替える
4. click event を保存する
5. スプレッドシートへ `自前 click 数` を反映する

## KPI シート方針

### 目的

- 運用KPIを `投稿管理 / 分析` とは別に一目で確認できるようにする
- 当面の目標である `合計インプレッション 50万` と `X→LPクリック率 5%` を常時見える化する

### KPI シートの最小構成

- `KPI`
- `目標値`
- `実績値`
- `進捗率`
- `判定`
- `メモ`
- `更新日時`

### 初期KPI

1. 累計インプレッション
- 目標値: `500,000`
- 実績値: `posts.latestImpressions` の累計

2. X→LPクリック率
- 目標値: `5.00%`
- 実績値: `累計LPクリック / 累計インプレッション`
- クリック数の正本は `clicks` table を使う

3. 累計LPクリック
- 目標値: `25,000`
- 実績値: `clicks` table の累計

### 同期ルール

- `syncToSpreadsheet` 実行時に `投稿管理 / 分析 / KPI / ネタ帳` をまとめて再計算する
- `投稿管理` も単票 upsert ではなく全置換を正本とし、削除済み投稿がシートに残らないようにする

## 候補保持ポリシー

### 方針

- `selected=true` の候補は残す
- `status=posted` の候補は残す
- `未使用 + 古い` 候補は 7日で削除する

### 未使用の定義

- `selected=false`
- `status != posted`

### 実装方針

- 日次バッチで `candidates` を走査する
- `updatedAt` が 7日より前の未使用候補を削除する
- 初期運用では `03:00 JST` に日次実行する
